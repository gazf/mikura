## ADR-018: Device ID based Liveness lock management

**Decision**: Generate and persist a permanent Device ID (UID, UUID v4) per client device, include it in JWT authentication for verification, and identify locks by `(userId, deviceId)`. TTL is extended via WSS heartbeat; on failure the lock is auto-released via Deno KV `expireIn`; on graceful shutdown it is released immediately via WSS terminate.

**Model**:

- **Device ID (UID)**: generated by the client on first launch, persisted in Local AppData, unchanged across restarts and reinstalls
- **JWT**: issued by the server on authentication, includes deviceId in the `device` claim, valid for 1 hour
- **Lock**: identified by `(userId, deviceId, path)`, TTL 30 seconds, auto-deleted via Deno KV `expireIn`
- **WSS connection**: extends TTL via heartbeat (10-second interval, payload is just deviceId); also the channel for real-time notifications

**Flow**:

```
[First launch]
1. Client: check device.json in the same directory as the executable
   - If missing, generate UUID v4 and save
   - If present, load it

[Login]
2. POST /auth/login { username, password, deviceId }
3. Server: authenticate -> register/update device table -> issue JWT (with device claim)

[WSS connection]
4. Client: ws://server/events?token=<JWT>
5. Server: verify JWT (check device claim against the connection source)

[File open]
6. POST /locks/* (Bearer JWT, X-Device-Id header)
7. Server: identify by deviceId, store lock (expireIn: 30s)
8. Server: broadcast lock_acquired to all clients via WSS

[Heartbeat]
9. Client: send { type: "heartbeat", deviceId } over WSS every 10 seconds
10. Server: extend TTL of all locks tied to deviceId by 30 seconds

[File close]
11. Client: upload (if edited)
12. Client: DELETE /locks/* (X-Device-Id header)
13. Server: confirm deviceId match -> delete lock
14. Server: broadcast lock_released via WSS

[Graceful shutdown]
15. Client: send { type: "terminate", deviceId } over WSS
16. Server: delete all locks tied to deviceId + broadcast

[Abnormal shutdown]
17. WSS disconnect detected -> server does nothing (optional)
18. After 30 seconds -> each lock auto-deleted by KV expireIn
19. In-house sweeper detects TTL expiry -> broadcast lock_released
```

**Schema**:

```typescript
// Deno KV
["devices", deviceId] -> {
    deviceId: string,
    userId: number,
    label: string,         // e.g. "OFFICE-PC-01"
    firstSeenAt: string,
    lastSeenAt: string,
    ipAddress?: string,
}

["devices-by-user", userId, deviceId] -> true  // reverse lookup

["locks", filePath] -> {
    userId: number,
    deviceId: string,      // device holding the lock
    acquiredAt: string,
    expiresAt: string,     // informational; actual TTL is KV expireIn
}
```

**JWT claims**:

```json
{
    "sub": 42,
    "device": "uuid-v4-here",
    "iat": 1745568000,
    "exp": 1745571600
}
```

**Persistence**:

- Storage location: `device.json` in the same directory as the client executable (directly under `AppContext.BaseDirectory`)
- The unit of the ID is the **install unit**. The same ID applies even when users switch on the same PC; copying the folder to another PC migrates the ID (unintended ID sharing via roaming profiles is structurally impossible)
- Even with single-file publish (`PublishSingleFile`), `AppContext.BaseDirectory` returns the executable's location, so it remains stable
- Placement note: do not place under a non-writable location like `Program Files`. Assume a portable placement (a user-writable folder)

**Handling of the same user across different devices**:

Identification is by `(userId, deviceId)`, but **takeover by a different deviceId is allowed for the same userId**:

```typescript
async function acquireLock(filePath, userId, deviceId): LockResult {
    const existing = await kv.get<LockData>(Keys.lock(filePath));
    
    if (existing.value) {
        if (existing.value.userId === userId) {
            // same user -> renew (same deviceId) or takeover (different deviceId)
            const updated = { 
                userId, 
                deviceId,  // overwrite with new deviceId
                acquiredAt: existing.value.acquiredAt,
                expiresAt: new Date(Date.now() + LOCK_TTL_MS).toISOString(),
            };
            await kv.set(key, updated, { expireIn: LOCK_TTL_MS });
            return { success: true, lock: updated };
        }
        // different user -> reject
        return { success: false, reason: "locked_by_other_user" };
    }
    
    // new acquisition
    return createLock(filePath, userId, deviceId);
}
```

Same approach as OneDrive ("take back a file you opened on your PC from your laptop" is possible).

**Rationale**:

- **ID persistence**: deviceId is unchanged across reconnects and restarts -> no need for complex logic like "session_restore"
- **Integration with JWT**: deviceId is verified as a JWT claim, preventing forgery
- **Easy audit tracking**: "when, from which device, what was done" can be tracked permanently per device
- **Same user / different device support**: naturally distinguished by deviceId comparison
- **Lightweight heartbeat**: payload is just deviceId, fixed size
- **Management UI achievable (future)**: list of "currently logged-in devices", forced logout, etc.

**Load estimation at 100-user scale**:

- Lock acquisition: ~5ms per op, peak 50 ops/sec, within Deno KV capacity
- Heartbeat: 100 users x 10-second interval = 10 ops/sec, negligible
- broadcast: WSS send to 100 users, lightweight

**Recovery from abnormal shutdown**:

- WSS disconnect -> KV TTL expires in 30 seconds -> locks auto-expire
- UX close to Samba (immediate expiry on TCP disconnect), realized over HTTPS

**Rejected alternatives**:

- **Client-generated session ID (changes on WSS disconnect)**: complex handover on reconnect
- **Server-issued session ID**: requires a complex session_restore protocol
- **Heartbeat sends the list of held locks**: oversized payload, violates SSOT principle
- **Pure TTL only**: cannot detect failures immediately
